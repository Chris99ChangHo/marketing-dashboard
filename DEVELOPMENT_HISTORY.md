# Development History & Troubleshooting

이 문서는 본 마케팅 대시보드를 구축하면서 마주한 극한의 인프라 제약과 이를 극복한 엔지니어링 여정을 기록합니다.

## 📖 PDF 리포트 자동화: 5전 6기의 여정

대시보드 데이터를 PDF로 변환하는 기능은 이 프로젝트의 핵심 비즈니스 가치였습니다. 하지만 **cPanel 공유 호스팅**이라는 제약(OS 수준의 패키지 매니저 `apt-get` 사용 불가, 커스텀 바이너리 설치 제약)은 엄청난 난관이었습니다. 이를 해결하기 위한 5번의 실패와 6번째의 성공 과정은 다음과 같습니다.

1. **Playwright (1차 시도 - 실패)**: 최신 웹 기술인 헤드리스 브라우저를 통한 렌더링을 시도했으나, 브라우저 바이너리 설치 권한 부족으로 실패.
2. **WeasyPrint (2차 시도 - 실패)**: 파이썬 기반의 강력한 PDF 라이브러리지만, 핵심 C 의존성 라이브러리(Pango, Cairo)를 공유 호스팅 환경에 설치할 수 없어 실패.
3. **ReportLab (3차 시도 - 보류)**: 설치는 가능하나, 순수 코드로 좌표 기반의 레이아웃을 그려야 하므로 웹(HTML/CSS) 기반의 풍부한 UI를 그대로 옮기는 데에 표현력 한계 및 생산성 저하.
4. **FPDF2 (4차 시도 - 보류)**: ReportLab과 마찬가지로 복잡한 차트와 웹 레이아웃을 표현하기에 한계 존재.
5. **PDFKit / wkhtmltopdf (5차 시도 - 실패)**: 역시 실행 가능한 시스템 바이너리를 요구하여 권한 문제에 부딪힘.

### 🏆 6차 시도: `xhtml2pdf` + 서버 사이드 차트 렌더링 (성공!)
결국 시스템 의존성 없이 순수 파이썬(Pure Python)으로 동작하는 **`xhtml2pdf`**를 채택했습니다. 
하지만 `xhtml2pdf`는 자바스크립트(Chart.js)를 실행하지 못한다는 치명적인 단점이 있었습니다. 

이를 극복하기 위해 **하이브리드 렌더링 방식**을 고안했습니다:
- **데이터 분석 및 시각화**: 백엔드에서 `Matplotlib`과 `Seaborn`을 사용해 데이터를 분석하고 차트 이미지를 생성합니다.
- **Base64 인코딩 주입**: 생성된 차트를 `Base64` 문자열로 변환한 뒤, 서버 사이드 템플릿(Jinja2)의 HTML `<img>` 태그에 직접 주입합니다.
- **최종 변환**: `xhtml2pdf`가 이렇게 렌더링된 순수 HTML(+Base64 이미지)을 읽어들여 픽셀 퍼펙트(Pixel-Perfect)한 브리핑용 PDF를 생성합니다. 
- **결과**: 인프라 한계를 완벽히 우회하면서도 매우 강력하고 아름다운 시각화 리포트를 완성할 수 있었습니다.

## 🛠 인프라 제약 극복과 프로세스 킬링(Signal 15) 선제적 방어

공유 호스팅 환경은 리소스 모니터링이 매우 엄격하여, 조금이라도 한도를 넘으면 프로세스가 강제 종료됩니다.

### 1. Rate Limiting과 Memory Storage 우회
API 남용을 막기 위해 `Flask-Limiter`를 도입했으나, 통상적으로 쓰이는 Redis를 호스팅 환경에서 사용할 수 없었습니다. 그 결과 500 에러가 빈발했습니다. 이를 해결하기 위해 백엔드 구조를 분석하고, 단일 프로세스 환경에 맞춘 인메모리 스토리지(`memory://`)로 Limiter 설정을 변경함으로써 문제를 우회 및 완벽히 해결했습니다.

### 2. 프로세스 킬링(SIGTERM, Signal 15) 완벽 차단
서버 사이드 차트 렌더링을 위해 `Matplotlib`, `NumPy`, `OpenBLAS`를 도입한 직후, 서버(Passenger WSGI)가 자주 강제 종료(Kill)되는 치명적인 문제가 발생했습니다. 

- **원인 분석**: `OpenBLAS`와 같은 고성능 수치 연산 라이브러리가 연산 속도 향상을 위해 시스템의 모든 CPU 코어 수만큼 멀티 스레드를 무단으로 생성했고, 이것이 cPanel의 공유 리소스 스레드 제한(Thread Limit)을 초과하여 SIGTERM을 유발한 것입니다.
- **해결 및 최적화**: 애플리케이션 진입점인 `passenger_wsgi.py`의 가장 상단에, 즉 문제가 되는 라이브러리들이 메모리에 로드되기 직전에 멀티 스레딩을 강제로 억제하는 환경 변수를 주입했습니다.
  
  ```python
  import os
  # NumPy, OpenBLAS 등이 멀티스레드를 생성하여 리소스 초과를 일으키는 것을 원천 차단
  os.environ['OPENBLAS_NUM_THREADS'] = '1'
  os.environ['GOTO_NUM_THREADS'] = '1'
  os.environ['OMP_NUM_THREADS'] = '1'
  ```

이러한 **OS 수준의 선제적 최적화**를 통해 리소스 초과로 인한 서버 다운 현상을 완벽히 방어하고, 제한적인 공유 호스팅 환경 위에서도 견고하게 동작하는 안정적인 파이프라인을 구축해 냈습니다.
